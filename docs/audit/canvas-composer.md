# Canvas / Composer Audit

**Date:** 2026-07-07 · **Scope:** `Views/ComposerView.swift`, `Views/DisplayCanvasView.swift`, `Models/ImageProcessor.swift`, `Models/DevicePreset.swift`, `Models/DisplayDevice.swift`, `Tests/ToneCompressionTests.swift`, `Tests/BLEChunkSizeTests.swift`

## Overview

The composer lets a user place a photo on a fixed-aspect canvas matching the connected e-paper panel, pan/zoom-crop it, add draw/text/QR annotations, apply e-ink-oriented adjustments, and render + dither + pack the result for BLE upload. Overall code health is good: the render pipeline is value-snapshot-based and off-main-thread, preview work runs on a downscaled copy behind a stale-result token, and the send path validates packed byte counts against the live device config. The main weaknesses are concentrated in gesture arbitration (a single `simultaneousGesture` pinch that fights the ever-present drag gesture) and in coordinate-space handling when the canvas box changes size (rotation), plus some dead code accumulated from pipeline refactors.

## Findings

### [HIGH] Rotation resets gesture baselines but not the transform — photo jumps on the next drag/pinch
- **Location:** `Views/ComposerView.swift:155-163` (layout branch), `Views/DisplayCanvasView.swift:61-63, 116`
- **Issue:** `ComposerView.body` switches between `portraitLayout` and `landscapeLayout` via `if verticalSizeClass == .compact`. The two branches are distinct structural identities, so rotating the phone destroys and recreates `DisplayCanvasView`, resetting its internal `@State` gesture baselines to defaults (`basePan = .zero`, `baseScale = 1`). The actual transform (`pan`/`scale`) lives in `ComposerView` bindings and survives. The next drag then computes `pan = basePan(.zero) + translation` and the next pinch computes `scale = max(1, 1 × value)` — snapping the photo back toward the untransformed state and discarding the user's crop. This is exactly the stale-baseline class of bug `transformResetToken` was built to solve, but rotation bypasses that mechanism (the token doesn't change, and even if it did, the new view's baselines should be *seeded from* `pan`/`scale`, not zeroed).
- **Impact:** Anyone who pans/zooms a photo, rotates the phone, and touches the canvas again loses their framing with no undo (photo transform is not on the undo stack).
- **Recommendation:** Either (a) capture gesture baselines at gesture *start* (`onChanged` first fire: `basePan = pan; baseScale = scale`) instead of persisting them across gestures, which eliminates the whole baseline-sync problem including `transformResetToken`; or (b) seed `basePan`/`baseScale` from the bound `pan`/`scale` in `onAppear`. Option (a) is the structural fix. Also consider giving the canvas a stable identity (`.id`) or extracting the shared layout so rotation doesn't recreate it (mid-stroke state `currentStroke` is also dropped today).

### [HIGH] Pinch and drag gestures run simultaneously and fight each other
- **Location:** `Views/DisplayCanvasView.swift:111` (`.simultaneousGesture(zoomGesture)`), `186-194` (interaction layer), `367-391` (`zoomGesture`), `393-410` (`drawGesture`), `416-454` (`selectionGesture`)
- **Issue:** `MagnificationGesture` is attached to the container with `.simultaneousGesture`, while the transparent interaction layer carries a `DragGesture(minimumDistance: 0)` in every mode. `simultaneousGesture` explicitly opts both into running together, and SwiftUI's `DragGesture` keeps reporting first-touch translation while a second finger pinches. Concrete failure modes:
  1. **Pinch-zoom drifts the photo:** in `.move` mode with nothing selected, a pinch both scales (`zoomGesture`) and pans (`selectionGesture`'s else-branch) — the crop wanders diagonally while zooming.
  2. **Pinch over an element moves it:** `selectionGesture` hit-tests `v.startLocation` regardless of selection, so pinching with a finger starting on a text/QR item drags that item once translation exceeds the 10pt slop — even when the user only intended to zoom the photo (element not selected) or resize the element in place (element selected, which is documented as resizing "around its fixed center").
  3. **Double undo entries:** when pinch-resizing a selected element, `selectionGesture.onEnded` commits `preDragSnapshot` *and* `zoomGesture.onEnded` commits `resizeSnapshot` — two near-identical undo entries for one interaction, so Undo appears to do nothing on first tap.
  4. **Pinch in Draw mode lays a stray stroke:** `drawGesture` (`minimumDistance: 0`) records the first finger's path during a pinch and commits it as a stroke (`points.count > 1` passes for any real pinch), while the photo also zooms underneath.
- **Impact:** Core interaction quality: jumpy crops, elements that scoot while being resized, phantom strokes, and confusing undo behavior. Affects every user on every session.
- **Recommendation:** Arbitrate explicitly: suppress the drag path once a magnification is in progress (e.g. a `@GestureState`/flag set in `zoomGesture.onChanged` that makes `selectionGesture`/`drawGesture` bail and revert), or replace the simultaneous composition with `ExclusiveGesture`/`highPriorityGesture` per mode. If the project's intended design (per prior planning notes) is corner-handle resize with pinch reserved exclusively for photo zoom, implementing that would remove the selected-element branch from `zoomGesture` and most of this conflict surface. Fold the two undo commits into one.

### [HIGH] Annotations and pan are stored in absolute canvas points and never remapped when the canvas box changes size
- **Location:** `Views/DisplayCanvasView.swift:114-115` (`canvasSize` tracking), `Views/ComposerView.swift:1143-1198` (`renderComposite`)
- **Issue:** Stroke points, text/QR positions, element sizes, and `pan` are all in on-screen canvas points. When the canvas box size changes — device rotation (portrait canvas width ≠ landscape canvas width), iPad multitasking resize, or the panel config arriving mid-session and changing `displaySize`'s aspect ratio — nothing rescales this state. The photo's *relative* framing mostly survives (aspect-fill base scale is relative) but `pan` is absolute, and every annotation stays at its old point coordinates inside a differently-sized box. `renderComposite` then maps points → pixels with `k = width / canvasSize.width` using the *current* box, so the send faithfully reproduces the now-rearranged screen — the user's original composition is silently lost, not just displayed wrong.
- **Impact:** Rotating the phone (or receiving a late config with a different resolution — see the Medium finding below) visibly shifts text/QR/strokes relative to the photo and canvas. Compounded by the baseline-reset bug above, rotation is currently destructive to a composition.
- **Recommendation:** Store annotation geometry and pan in normalized canvas space (0–1 fractions of the box, with sizes as a fraction of box width) or directly in panel-pixel space, converting to points only for display. As a cheaper stopgap, rescale all point-space state by `newBox / oldBox` inside the `.onChange(of: box)` handler.

### [MEDIUM] `ImageProcessor.preview` misses the empty-pixel-buffer guard that `process`/`processWithPreview` have — out-of-bounds crash path
- **Location:** `Models/ImageProcessor.swift:141-142` (compare `:111-112` and `:170-171`)
- **Issue:** `preview()` calls `rgbaPixels(from:width:height:)` and immediately feeds the result to `toFloat(pixels, count: width * height)` without the `guard !pixels.isEmpty` present in the two sibling entry points. `rgbaPixels` returns `[]` whenever `CGContext` creation fails (zero/negative dimensions, extreme allocation pressure). `toFloat` then indexes `pixels[i*4+0]` for `width*height` pixels → index-out-of-range crash.
- **Impact:** Low probability, but a hard crash on the main preview path when it fires (e.g. a stale saved entity with 0 dimensions, or memory pressure on huge panels). Since `generatePreview` may run with `device?.config == nil` and fall back to entity dimensions, the zero-size case is not purely theoretical.
- **Recommendation:** Add the same `guard !pixels.isEmpty else { return nil }` to `preview()`. Consider also folding `process`/`preview`/`processWithPreview` into one implementation (see dead-code finding) so the guard can't diverge again.

### [MEDIUM] Composing against a stale fallback resolution while `device.config` is still loading
- **Location:** `Views/ComposerView.swift:125-133` (`displayWidth`/`displayHeight` fallback), `:200-205` (config `.onChange`), `:1061-1080` (`generatePreview`), `:1083-1088` (`sendPhoto` guard)
- **Issue:** The send path is correctly gated: `sendPhoto` refuses to encode when `device.config == nil` (fails the upload with an explanatory message and triggers `readConfig()`), and `ODDevice.uploadImage` re-validates the packed byte count against the live config — good. But everything *before* send trusts the cached `entity.width/height/colorScheme`: the canvas aspect ratio, the palette swatches, smart dithering choice, and the Preview sheet all render at the fallback values. If the cached entity is stale or was saved with the 800×480/scheme-0 default (ContentView's add flow falls back to `800×480, scheme 0` when config wasn't loaded at save time), the user composes against the wrong aspect ratio and palette. When the real config arrives mid-session, `displaySize` snaps to the true aspect — and because of the point-space finding above, existing annotations/pan are not remapped, scrambling whatever was already placed. Additionally, the config-blocked send is a dead end: the user sees "Waiting for a valid device configuration…" and must notice, dismiss, and manually retry after the background `readConfig()` completes.
- **Impact:** Composer-side symptom of the broader add-device config race (audited separately). Worst case is a user who composes an entire layout on a wrong-aspect canvas, or whose layout rearranges the moment config lands.
- **Recommendation:** Surface a lightweight "reading display configuration…" state on the canvas while `device.config == nil` and the cached entity has never been confirmed against hardware; on the send guard, auto-retry once `readConfig` succeeds instead of requiring a manual retry. The annotation-remap fix above removes the mid-session scramble.

### [MEDIUM] QR pinch-resize regenerates the QR through Core Image on the main thread every frame
- **Location:** `Views/DisplayCanvasView.swift:171-182` (`qrOverlays`), `:333-342` + `:367-391` (live size mutation during pinch), `:507-545` (`odGenerateQR` / cache)
- **Issue:** The `odQRCache` key includes the rounded integer size, so while pinch-resizing a QR item every ~1px size change is a cache miss that runs `CIFilter.qrCodeGenerator` + `falseColor` + a `CIContext.createCGImage` rasterization synchronously inside the SwiftUI render pass. A pinch across the 40–300pt range creates a few hundred distinct cache entries and a few hundred main-thread CI renders in a second or two.
- **Impact:** Frame drops / jumpy pinch on older devices, especially with long QR payloads (denser modules); transient memory growth in the cache (NSCache will evict under pressure, but only under pressure).
- **Recommendation:** During an active resize, render the *cached* base QR image scaled via the view's `.frame` (interpolation `.none` keeps it acceptably crisp) and regenerate at the final size once in `onEnded`; or quantize the cache key to coarser buckets (e.g. 8px) during gestures. The composite/send path is unaffected (it renders once, off-main).

### [LOW] Photo transform is under-constrained: pan is unclamped and zoom has no upper bound
- **Location:** `Views/DisplayCanvasView.swift:379` (`scale = max(1, baseScale * value)`), `:434-437` (pan update)
- **Issue:** `scale` is floored at 1 but unbounded above; `pan` is completely unclamped. A user can push the photo entirely off-canvas (leaving pure white) or zoom a low-res source photo hundreds of times past its native resolution with no feedback. There is no "photo must cover the canvas" constraint typical of croppers, so aspect-fill guarantees erode as soon as pan/zoom are touched.
- **Impact:** Easy to send a blank or unrecognizably-cropped image by accident; nothing warns that the panel will show white. Also makes the composite render draw an arbitrarily huge `CGRect` (harmless but wasteful).
- **Recommendation:** Clamp pan in `onChanged`/`onEnded` so the scaled image always covers the box (standard crop-box math: `|pan| ≤ (drawSize − box)/2` per axis), and cap `scale` at something like 8–10× or a source-pixel-derived limit.

### [LOW] Live canvas is not colorimetrically what gets sent (mostly by design, two genuine gaps)
- **Location:** `Views/ComposerView.swift:510-531` (color-mode override copy), `:936-953` (`refreshCanvasImage` tone pass), `:1101-1117` (send pipeline)
- **Issue:** The dithered/quantized result is intentionally only shown via Preview and the send overlay, and the color-mode panel explicitly warns that overriding the scheme only affects the on-screen palette. Two residual mismatches are worth knowing about: (1) with "Measured palette" on, the live canvas applies tone compression to the *photo only*, while the send/Preview pipeline compresses the *entire composite* — annotations and the white background shift slightly between canvas and panel; (2) annotation colors picked from an overridden palette (e.g. green on a B/W/R panel) are silently nearest-color-quantized at send time — the warning text covers the photo but a user may not expect their green text to come out black.
- **Impact:** Minor preview-fidelity surprises for users of the expert override or measured palette; Preview (the authoritative check) is accurate in both cases because `generatePreview` and `sendPhoto` share the same pipeline and read the hardware scheme when available.
- **Recommendation:** Accept (1) as a documented approximation or run `compressTone` over annotations in the canvas too; for (2), restrict annotation swatches to the *hardware* scheme's palette (the override already claims to be display-only, so letting it change authoring swatches is arguably the bug).

### [LOW] Dead code and pipeline duplication in `ImageProcessor` and models
- **Location:** `Models/ImageProcessor.swift:103-131` (`process`), `:428-436` (`pack1bpp` `invert:` parameter), `Models/DevicePreset.swift` (entire file), `Views/DisplayCanvasView.swift:549-568` (`CanvasMode` `title`/`systemImage`/`CaseIterable`), `Views/ComposerView.swift:1062` (`refreshCanvasImage()` call in `generatePreview`)
- **Issue:** `ImageProcessor.process(image:...)` has zero callers — `sendPhoto` uses `processWithPreview` and the sheet uses `preview`, each duplicating the same 15-line pipeline (which is how the missing guard in `preview` diverged). `pack1bpp`'s `invert` parameter is always `false`. `DevicePreset` (including its `.all` catalog) is referenced by no other file. `CanvasMode.title`/`systemImage`/`CaseIterable`/`Identifiable` are unused since the tool-chip refactor (`ComposerTool` owns UI metadata now). `generatePreview` calls `refreshCanvasImage()` although the preview renders from `baseImage`, not `canvasImage`.
- **Impact:** Maintenance drag and drift risk (already materialized once as the `preview` guard divergence); no runtime effect.
- **Recommendation:** Delete `process` (or reimplement `preview`/`processWithPreview` on top of a single core), drop the `invert` parameter, remove `DevicePreset` if the add-device flow really no longer offers presets, and trim `CanvasMode` to the bare enum.

### [LOW] Test coverage gap: wire-format packing and transform math are unguarded
- **Location:** `Tests/ToneCompressionTests.swift`, `Tests/BLEChunkSizeTests.swift` vs `Models/ImageProcessor.swift:414-511`
- **Issue:** Existing tests pin the tone-compression port to its Rust reference and the config chunk size to the bundled JS — both good. But nothing covers the packing functions (`pack2planes` plane semantics where yellow deliberately does not set the BW plane, `packGray4`'s LUT, `scheme4Remap`'s palette→wire reordering, row padding for non-multiple-of-8 widths like 296×128) or `expectedPackedByteCount` agreement with actual packed sizes, and nothing covers the screen→pixel transform in `renderComposite`. These encode hardware wire contracts that would fail silently as scrambled panel output.
- **Recommendation:** Add table-driven tests asserting packed bytes for tiny synthetic images per scheme (including an odd width), and a property test that `pack(...)_count == expectedPackedByteCount` for every scheme. A `renderComposite` golden test at small size would pin the crop math.

### [LOW] Convention drift: planned corner-handle resize vs shipped pinch-resize
- **Location:** `Views/DisplayCanvasView.swift:367-391`, `Views/ComposerView.swift:445, 475` (UI copy "pinch to resize")
- **Issue:** Project planning notes specify that elements should resize via corner drag handles with pinch reserved exclusively for photo zoom/crop. The shipped code implements pinch-resize on the selected element (and documents it in the selection-chrome comment and panel hint text); no corner-handle code exists. This is informational — but the pinch-vs-drag arbitration problems in the HIGH gesture finding are a direct consequence of overloading pinch.
- **Impact:** None by itself; it means the intended interaction model hasn't landed.
- **Recommendation:** Decide which model is canonical. If corner handles are still the plan, implementing them resolves most of the gesture-conflict finding for free; if pinch-resize is now canonical, update the planning notes.

## Positive observations

- **Off-main render pipeline done right:** `compositeSnapshot()` captures all `@State` as value types on the main thread, `renderComposite` is pure over the snapshot, and the dither+pack runs on a background queue with the overlay shown immediately — no SwiftUI state touched off-main.
- **Preview/send fidelity:** `processWithPreview` runs the dither exactly once and returns both the wire bytes and the preview bitmap, so the progress-reveal image is pixel-identical to what the panel receives. `generatePreview` and `sendPhoto` read the color scheme from the live config at render time, avoiding stale-`@State` encodes.
- **Send safety rails:** `sendPhoto` hard-blocks on missing config rather than guessing a resolution, and `ODDevice.uploadImage` independently validates the packed byte count against `expectedPackedByteCount` before transmitting.
- **Memory discipline:** live adjustments run over a ≤1600px preview copy on a serial queue with a stale-result token (`canvasRenderToken`), preventing the classic slider-drag Core Image memory spike; full resolution is used only for the one-shot send render.
- **Thoughtful small fixes with documented rationale:** the `photoItem = nil` re-pick trick, the CGImage-backed QR rendering (with cache), the `transformResetToken` baseline sync (correct for its designed cases — see HIGH finding for the rotation gap), and undo snapshots that preserve element identity.
